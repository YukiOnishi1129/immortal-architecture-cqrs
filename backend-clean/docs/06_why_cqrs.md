```
┌─────────────────────────────────────────────┐
│  コンビニ                                    │
│                                              │
│  🏬 商品棚（読む場所）     💰 レジ（書く場所）│
│  ┌─────────────┐         ┌──────────────┐   │
│  │ おにぎり 🍙  │         │ バーコード    │   │
│  │ パン 🍞     │         │ ピッ！       │   │
│  │ お茶 🍵     │         │ 在庫 -1      │   │
│  │ 弁当 🍱     │         │ 売上 +500円  │   │
│  └─────────────┘         └──────────────┘   │
│                                              │
│  → パッと見て選ぶだけ      → 正確に処理する   │
│  → 速さが大事！           → 正しさが大事！    │
└─────────────────────────────────────────────┘
```

**商品棚（読む）** と **レジ（書く）** は **求められることが全然違う**：

| | 商品棚（読む） | レジ（書く） |
|---|---|---|
| **目的** | お客さんが商品を探す | 買ったものを正確に記録する |
| **大事なこと** | 見やすい・探しやすい・速い | 正しい・漏れがない |
| **頻度** | めっちゃ多い（みんな見る） | 少なめ（買う人だけ） |

コンビニが「レジで商品も見せます！」ってやったら大混雑しますよね？
**だから分けてるんです。**

## 💻 今のbackend-cleanの「読む」を見てみよう

### 🔍 ノート一覧を取得するSQL（現状）

今の `ListNotes` クエリを見てください：

```sql
SELECT
    n.*,
    t.name AS template_name,        -- テンプレート名も欲しい
    a.first_name,                   -- オーナーの名前も欲しい
    a.last_name,
    a.thumbnail AS owner_thumbnail  -- アイコンも欲しい
FROM notes n
JOIN templates t ON t.id = n.template_id   -- ← JOIN 1つ目
JOIN accounts a ON a.id = n.owner_id       -- ← JOIN 2つ目
WHERE ...
ORDER BY n.updated_at DESC;
```

さらに、ノート1件を表示するときは **sections（本文）** も必要なので：

```sql
SELECT s.*, f.label, f."order", f.is_required
FROM sections s
JOIN fields f ON f.id = s.field_id    -- ← JOIN 3つ目
WHERE s.note_id = $1;
```

### 😰 何が問題なの？

**今は小さいアプリだから大丈夫。でも大きくなると…**

```
【今】ノート100件、ユーザー10人

  GET /notes → 3テーブルJOIN → 0.05秒 ✅ 余裕！

──────────────────────────────────────────

【将来】ノート100万件、ユーザー10万人

  GET /notes → 3テーブルJOIN → 2.5秒 🔥 遅い！
  しかも同時に100人がアクセス → DBパンク 💀
```

#### 問題をまとめると：

| # | 問題 | なんで起きる？ |
|---|------|--------------|
| **1** | 🐌 **読み取りが遅い** | 毎回JOINして毎回計算してるから |
| **2** | 🏋️ **DBに負荷集中** | 読む人も書く人も同じテーブルに殺到 |
| **3** | 🔒 **読み書きが邪魔し合う** | 書き込みのロックで読み取りが待たされる |
| **4** | 🧩 **読む用のデータ形式と書く用が違う** | JOINで無理やり合わせてる |

## 🤔 でも待って、本当に問題？

**大事なこと：今のCRUDで問題ないなら、CQRSは不要です。**

```
┌──────────────────────────────────────────┐
│  🚗 CRUDで十分なケース                    │
│                                           │
│  ✅ ユーザー数が少ない（数百〜数千人）      │
│  ✅ データ量が少ない（数万件以下）          │
│  ✅ 読み書きの比率が同じくらい              │
│  ✅ リアルタイム性が求められない            │
│  ✅ 管理画面・社内ツール                   │
│                                           │
│  → JOINで十分！CQRSは過剰！              │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│  🏎️ CQRSを検討すべきケース                │
│                                           │
│  ⚠️ 読み取りが圧倒的に多い（読み:書き=9:1）│
│  ⚠️ データ量が多い（数十万件〜）           │
│  ⚠️ 一覧画面の表示が遅くなってきた         │
│  ⚠️ 読む形と書く形が全然違う              │
│  ⚠️ toC向けサービス（ユーザー数が多い）    │
│                                           │
│  → CQRS検討の価値あり！                   │
└──────────────────────────────────────────┘
```

### 📊 実は世の中のWebアプリの読み書き比率

ほとんどのWebアプリは**読み取りが圧倒的に多い**です。

```
  SNS（Twitter/Xとか）:    読み 99% ████████████████████ : 書き 1%  ▏
  ECサイト:               読み 95% ███████████████████  : 書き 5%  █
  ノートアプリ（これ）:     読み 80% ████████████████     : 書き 20% ████
  管理画面:               読み 50% ██████████           : 書き 50% ██████████
```

つまり、**ほとんどのアプリで「読み取りの最適化」は意味がある**んです。

## 💡 CQRSってなに？

**CQRS = Command Query Responsibility Segregation**
日本語にすると **「コマンドとクエリの責務分離」**

もっと簡単に言うと：

> **「書く仕組み」と「読む仕組み」を分けましょう**

```
┌──────────────────────────────────────────────────┐
│  今のCRUD（ぜんぶ一緒）                            │
│                                                    │
│         📱 画面                                    │
│          │                                         │
│          ▼                                         │
│     ┌─────────┐                                    │
│     │ UseCase │  ← 読むのも書くのも同じ場所          │
│     └────┬────┘                                    │
│          │                                         │
│          ▼                                         │
│     ┌─────────┐                                    │
│     │   DB    │  ← 1つのテーブルで読み書き両方       │
│     │  notes  │                                    │
│     │templates│                                    │
│     │accounts │                                    │
│     └─────────┘                                    │
└──────────────────────────────────────────────────┘

              ↓ CQRSにすると ↓

┌──────────────────────────────────────────────────┐
│  CQRS（読み書き分離）                               │
│                                                    │
│         📱 画面                                    │
│        ／    ＼                                    │
│       ▼        ▼                                   │
│  ┌────────┐  ┌────────┐                            │
│  │Command │  │ Query  │                            │
│  │(書く)   │  │(読む)  │                            │
│  └───┬────┘  └───┬────┘                            │
│      │           │                                 │
│      ▼           ▼                                 │
│  ┌────────┐  ┌──────────────┐                      │
│  │  Write │  │  Read Model  │ ← 読む専用！          │
│  │   DB   │  │（キャッシュ    │   JOINなしで         │
│  │ (今の  │  │  テーブル）   │   一発取得！          │
│  │テーブル)│  └──────────────┘                      │
│  └───┬────┘         ▲                              │
│      │              │                              │
│      └──── 同期 ────┘                              │
│       書いたら読む用も更新                           │
└──────────────────────────────────────────────────┘
```

## 🍱 お弁当屋さんで考えるCQRS

もうちょっと身近な例で説明します。

### ❌ CQRS前：注文のたびに作る定食屋

```
お客さん「唐揚げ弁当ください」
→ 鶏肉切って（notes）
→ 衣つけて（templates）
→ 揚げて（JOIN）
→ ご飯よそって（JOIN）
→ 盛り付けて（JOIN）
→ 「お待たせしました！」（15分後）

次のお客さん「唐揚げ弁当ください」
→ また最初から全部やる…（15分後）
```

### ✅ CQRS後：作り置きのお弁当屋さん

```
【裏（Command側）】
朝、唐揚げ弁当を50個作り置きしておく
  → 新しい注文が入ったら、作り置きも更新

【表（Query側）】
お客さん「唐揚げ弁当ください」
→ はい、どうぞ！（3秒）

次のお客さん「唐揚げ弁当ください」
→ はい、どうぞ！（3秒）
```

**ポイント：読む側は「すでに完成品がある」状態にしておく**

## 🔧 具体的にこのプロジェクトでどう変わる？

### Before: 今のCRUD

ノート一覧を取得するたびに：

```
notes → JOIN templates → JOIN accounts → 結果を組み立て
  毎回3テーブルをガチャガチャ結合 🔧
```

### After: CQRS（キャッシュテーブル方式）

**書くとき（Command）**：
```
1. notesテーブルに書き込む（今まで通り）
2. ついでにnote_read_modelsテーブルも更新する ← NEW!
```

**読むとき（Query）**：
```
SELECT * FROM note_read_models WHERE ...
  → JOINなし！一発で全部取れる！🚀
```

### キャッシュテーブルのイメージ

```sql
-- 今のテーブル（正規化されてる = 書く用に最適）
notes          → id, title, template_id, owner_id, status
templates      → id, name, owner_id
accounts       → id, first_name, last_name, thumbnail
sections       → id, note_id, field_id, content
fields         → id, template_id, label, order

-- 新しく作るテーブル（非正規化されてる = 読む用に最適）
note_read_models → id, title, status,
                   template_name,           ← JOINしなくていい！
                   owner_first_name,        ← JOINしなくていい！
                   owner_last_name,         ← JOINしなくていい！
                   owner_thumbnail,         ← JOINしなくていい！
                   sections_json,           ← 全部入り！
                   created_at, updated_at
```

```
【Before】1回の読み取りで：
  notes JOIN templates JOIN accounts → sections JOIN fields
  = 4テーブル結合 + 2回のクエリ

【After】1回の読み取りで：
  SELECT * FROM note_read_models WHERE id = ?
  = 1テーブル + 1回のクエリ ← 圧倒的に速い！
```

## ⚖️ CQRSのメリット・デメリット

何でもそうだけど、**メリットとデメリットの両方を知ることが大事**。

### ✅ メリット

| # | メリット | 説明 |
|---|---------|------|
| **1** | 🚀 **読み取りが爆速** | JOINなしで一発取得 |
| **2** | 🏋️ **DBの負荷分散** | 読む用テーブルと書く用テーブルが別だから負荷が分散 |
| **3** | 🎯 **それぞれに最適化** | 読む用は検索しやすく、書く用は整合性重視 |
| **4** | 📈 **スケールしやすい** | 読み取りが増えても読む用だけ対応すればいい |

### ❌ デメリット

| # | デメリット | 説明 |
|---|----------|------|
| **1** | 📝 **コードが増える** | 書くときにRead Modelも更新する処理が必要 |
| **2** | ⏱️ **データの遅延** | 書いた直後に読むと古いデータが返る可能性（今回は同期更新なので大丈夫） |
| **3** | 🔄 **整合性の管理** | Write DBとRead Modelの同期を保つ仕組みが必要 |
| **4** | 🧠 **学習コスト** | チームメンバーがCQRSを理解する必要がある |

## 🏗️ 現場ではどうやって段階的に導入する？

**いきなりフルCQRSにする必要はない**です。段階的に進めるのが現場のやり方。

```
Step 1: 普通のCRUD（今のbackend-clean）
  └→ 十分速い。問題ない。

Step 2: 「一覧画面が遅い」が発生 🐌
  └→ まずインデックス追加で対応

Step 3: それでも遅い、JOINが重い
  └→ キャッシュテーブル（Read Model）を導入 ← 今回やること！
     └→ 同じPostgreSQL内にテーブルを追加

Step 4: さらにアクセスが増えてDBがパンク
  └→ Read ModelをRedisに移行
     └→ インフラコストと相談

Step 5: もっと大規模に
  └→ イベントソーシング + 非同期更新
     └→ ここまでいくのは超大規模サービスだけ
```

**今回はStep 3をやります。**
小〜中規模の現場で最もよくあるパターンです。

## 🎯 まとめ

| 質問 | 答え |
|-----|------|
| CQRSって何？ | 読む仕組みと書く仕組みを分けること |
| なんで必要？ | 読み取りが多いアプリで速度とスケーラビリティを確保するため |
| いつ必要？ | JOINが重い、一覧表示が遅い、読み書き比率が偏ってるとき |
| いつ不要？ | 小規模アプリ、管理画面、読み書きが同じくらいのとき |
| 今回どうやる？ | キャッシュテーブル方式（同じDB内にRead用テーブルを作る） |
| 現場ではどう？ | まずキャッシュテーブル → 必要になったらRedis → さらに必要ならイベントソーシング |
