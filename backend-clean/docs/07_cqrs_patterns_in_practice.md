# 🏭 現場のCQRS - Read Modelどこに置く？パターン比較

> 💡 **このドキュメントのゴール**
> 「CQRSのRead Modelってどこに置くの？」に対して、
> 現場で使われてるパターンを知って、**自分のプロジェクトで何を選ぶべきか**判断できるようになること。

---

## 🗺️ まず全体像を見よう

CQRSの「読む用のデータ（Read Model）」をどこに置くか、主に **4つのパターン** があります。

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Write DB（書く用）は全パターン共通で PostgreSQL等を使う  │
│                                                         │
│  Read Model（読む用）をどこに置く？                       │
│                                                         │
│  ┌───────────┐ ┌───────────┐ ┌──────────┐ ┌──────────┐ │
│  │① キャッシュ│ │② Redis    │ │③ Firebase│ │④ 検索    │ │
│  │  テーブル  │ │  /Memcached│ │ /Firestore│ │エンジン  │ │
│  │           │ │           │ │          │ │Elastic   │ │
│  │  同じDB内  │ │ 別のサーバー│ │  クラウド │ │Search等  │ │
│  └───────────┘ └───────────┘ └──────────┘ └──────────┘ │
│                                                         │
│  ← シンプル・安い          複雑・高い・高性能 →          │
│  ← 小〜中規模向け            大規模向け →               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

---

## ① キャッシュテーブル（同じDB内にRead用テーブル）

### 🍱 たとえ話

```
お弁当屋さんが、注文ノートとは別に
「今日のメニューボード」を店頭に置いてる感じ

  注文ノート（Write）= 正式な記録
  メニューボード（Read）= お客さんが見やすい形にまとめたもの

  同じ店の中にある！別の場所を借りたりしない！
```

### 📝 どういうこと？

同じPostgreSQLの中に、**読む専用のテーブル**を作る。

```sql
-- Write用（正規化されたテーブル、今まで通り）
notes, templates, accounts, sections, fields

-- Read用（非正規化されたテーブル、新しく追加）
note_read_models
  → JOINの結果を事前に保存しておく
  → SELECTするだけで全部取れる
```

### ✅ メリット

| メリット | 説明 |
|---------|------|
| 🏗️ **インフラ追加なし** | PostgreSQLだけで完結。Docker Composeも変更不要 |
| 💰 **コスト0** | 新しいサービスを契約しなくていい |
| 🔧 **シンプル** | SQLだけで完結。新しい技術を覚えなくていい |
| 🐛 **デバッグしやすい** | `SELECT * FROM note_read_models` で中身が見える |
| 🔄 **トランザクションで同期** | Write + Read Modelの更新を同じトランザクションにできる |

### ❌ デメリット

| デメリット | 説明 |
|----------|------|
| 🐌 **超高速ではない** | RDBなのでRedisほどは速くない（でも十分速い） |
| 🏋️ **同じDBに負荷** | 読み書きが同じDBサーバーに集中する |
| 📈 **スケール限界** | DBサーバー1台の性能が限界 |

### 🎯 こういう現場で使う

```
✅ スタートアップ（まだユーザー少ない）
✅ BtoB SaaS（ユーザー数千〜数万人）
✅ 社内システム
✅ 「まずCQRSを試したい」とき
✅ インフラにお金をかけたくないとき
```

### 💰 コスト感

```
追加コスト: ¥0（既存のDBを使うだけ）
```

## ② Redis / Memcached（インメモリキャッシュ）

### 🍱 たとえ話

```
お弁当屋さんの隣に、専用のショーケースを設置した感じ

  厨房（Write DB）= 作る場所
  ショーケース（Redis）= 完成品を並べておく場所

  別の場所（別サーバー）にある！
  電気が必要（お金がかかる）！
  でもガラス越しにパッと見える（超高速）！
```

### 📝 どういうこと？

**メモリ上**にデータを保存する別サーバーを立てる。
データはJSON等にシリアライズして保存。

```
Write: PostgreSQL（今まで通り）

Read: Redis
  key: "note:abc123"
  value: {"id":"abc123","title":"...","owner_name":"田中太郎",...}

  → キーを指定するだけで一瞬で取れる（1ms以下）
```

### ✅ メリット

| メリット | 説明 |
|---------|------|
| ⚡ **超高速** | メモリ上なので1ms以下で返せる |
| 🏋️ **DB負荷軽減** | 読み取りがDBに来なくなる |
| 📈 **スケールしやすい** | Redisクラスター（複数台）に分散可能 |
| ⏱️ **TTL（有効期限）** | 古いキャッシュを自動で消せる |

### ❌ デメリット

| デメリット | 説明 |
|----------|------|
| 💰 **お金がかかる** | AWS ElastiCache等の月額料金が発生 |
| 🔧 **インフラ増える** | Docker ComposeにRedis追加、接続管理 |
| 📦 **シリアライズ必要** | Go構造体 → JSON → Redis → JSON → Go構造体 |
| 💥 **揮発性** | サーバー再起動でデータ消える（再構築が必要） |
| 🔄 **整合性が難しい** | Write DBとRedisの同期タイミングのズレ |
| 🧠 **学習コスト** | Redisの知識が必要 |

### 🎯 こういう現場で使う

```
✅ toC向けサービス（ユーザー数万〜数十万人）
✅ レイテンシが重要（100ms以下必須）
✅ 読み取りが圧倒的に多い（95%以上が読み取り）
✅ セッション管理やランキング等にも使いたい
```

### 💰 コスト感

```
AWS ElastiCache (Redis):
  小さめ（cache.t3.micro）: 約 $15/月（≒ ¥2,200/月）
  中くらい（cache.r6g.large）: 約 $150/月（≒ ¥22,000/月）
  本番用（Multi-AZ）: 約 $300〜/月（≒ ¥44,000〜/月）
```

## ③ Firebase / Firestore（クラウドNoSQL）

### 🍱 たとえ話

```
お弁当屋さんが、UberEatsに出店した感じ

  厨房（Write DB）= 自分の店で作る
  UberEats（Firestore）= 別プラットフォームでお客さんに届ける

  自分で配達しなくていい（運用レス）！
  でも手数料がかかる！
  あとUberEatsの仕組みに合わせる必要がある！
```

### 📝 どういうこと？

GoogleのクラウドNoSQLデータベースにRead Modelを保存する。
**フロントエンドから直接読める**のが最大の特徴。

```
Write: PostgreSQL（バックエンドで書く）
  ↓ 同期
Read: Firestore（フロントエンドから直接読む）

  クライアント → Firestore → データ取得
  （バックエンドを経由しない！）
```

### ✅ メリット

| メリット | 説明 |
|---------|------|
| 🌐 **フロント直接アクセス** | バックエンドのAPIを通さず読める |
| 📡 **リアルタイム同期** | データ変更が自動でフロントに反映される |
| 🏗️ **インフラ管理不要** | Googleが面倒見てくれる（サーバーレス） |
| 📈 **自動スケール** | アクセス増えても勝手にスケール |

### ❌ デメリット

| デメリット | 説明 |
|----------|------|
| 💰 **従量課金** | 読み取り回数×単価。予測しにくい |
| 🔒 **ベンダーロックイン** | Googleに依存。乗り換えが大変 |
| 🧩 **クエリが弱い** | RDBのような複雑な検索は苦手 |
| 🔄 **同期が複雑** | PostgreSQL → Firestoreの同期パイプラインが必要 |
| 🔐 **セキュリティルール** | フロント直接アクセスはセキュリティ設計が難しい |

### 🎯 こういう現場で使う

```
✅ モバイルアプリ（オフライン対応もしたい）
✅ リアルタイムが必要（チャット、コラボ編集）
✅ フロントエンド主体のチーム
✅ インフラ管理したくないスタートアップ
```

### 💰 コスト感

```
Firestore:
  無料枠: 読み取り 50,000回/日、書き込み 20,000回/日
  超過分: 読み取り $0.06/10万回、書き込み $0.18/10万回
  小規模なら無料でいける。大規模だと予想外に高くなることも。
```

## ④ Elasticsearch等（検索エンジン）

### 🍱 たとえ話

```
お弁当屋さんが、食べログに掲載した感じ

  厨房（Write DB）= メニューを管理する場所
  食べログ（Elasticsearch）= お客さんが検索する場所

  「渋谷 唐揚げ 500円以下」みたいな検索ができる！
  でも掲載料がかかるし、情報の更新は手動！
```

### 📝 どういうこと？

全文検索エンジンにRead Modelを保存する。
**複雑な検索やあいまい検索**が得意。

```
Write: PostgreSQL
  ↓ 同期
Read: Elasticsearch
  → 「タイトルに"設計"を含むノートで、
     ステータスがPublishで、
     最近1週間に更新されたもの」
  → が爆速で返ってくる
```

### 🎯 こういう現場で使う

```
✅ 検索機能が重要なサービス（ECサイト、ナレッジベース）
✅ 全文検索が必要
✅ ファセット検索（絞り込み）が必要
```

### 💰 コスト感

```
AWS OpenSearch (Elasticsearch):
  最小構成: 約 $50/月〜
  本番用: 約 $200〜500/月
  → 一番高い選択肢
```

## 🤔 じゃあどれを選べばいいの？

### フローチャートで判断しよう

```
Start: 一覧表示が遅い？JOINが重い？
  │
  ├─ No → CQRSいらない。今のCRUDで十分 ✋
  │
  └─ Yes
      │
      ├─ まずインデックス追加で解決する？
      │   ├─ Yes → インデックスで解決 ✋
      │   └─ No ↓
      │
      ├─ 全文検索・あいまい検索が必要？
      │   ├─ Yes → ④ Elasticsearch
      │   └─ No ↓
      │
      ├─ フロントから直接読みたい？リアルタイム必要？
      │   ├─ Yes → ③ Firestore
      │   └─ No ↓
      │
      ├─ 秒間1000リクエスト以上？1ms以下が必要？
      │   ├─ Yes → ② Redis
      │   └─ No ↓
      │
      └─ ① キャッシュテーブル ← 大半のプロジェクトはここ！
```

### 🏢 現場でよくある進化パターン

```
Phase 1: CRUDで十分（スタートアップ初期）
  │
  │  ユーザー増えてきた...一覧が遅い...
  ▼
Phase 2: キャッシュテーブル導入 ← ★ 多くの現場はここで十分 ★
  │
  │  さらにアクセス増えた...DB限界...
  ▼
Phase 3: Redisに移行（Read Modelの置き場を変えるだけ）
  │
  │  検索機能がもっと必要に...
  ▼
Phase 4: Elasticsearch追加（検索専用）

  ※ Phase 2で止まってる現場がめっちゃ多い
  ※ Phase 3まで行くのは中〜大規模サービス
  ※ Phase 4は検索が重要なサービスだけ
```

## 🎓 このプロジェクトでは①を使う理由

```
┌─────────────────────────────────────────────┐
│  なぜ「キャッシュテーブル」を選ぶ？           │
│                                              │
│  1. 🎯 CQRSの本質（読み書き分離）に集中      │
│     → Redisの知識が前提にならない             │
│                                              │
│  2. 🔧 インフラがシンプル                     │
│     → PostgreSQLだけ。Docker Compose変更なし  │
│                                              │
│  3. 💰 現場で一番よくあるパターン             │
│     → 「これ知ってれば実務で使える」          │
│                                              │
│  4. 📈 発展しやすい                           │
│     → あとからRedisに差し替えるのも簡単       │
│     → Clean Architectureだからインターフェース│
│       の裏側を差し替えるだけ！                │
│                                              │
│  5. 🐛 デバッグしやすい                       │
│     → SQLで直接見れるから学習に最適           │
└─────────────────────────────────────────────┘
```

特にポイント **4** が大事。
Clean Architectureの **ポート（インターフェース）** の裏側を差し替えるだけで、
キャッシュテーブル → Redis に移行できます。

```
┌─────────────────────────────────────────────┐
│  NoteQueryRepository（インターフェース）       │
│    └─ Get(id) → NoteReadModel               │
│    └─ List(filters) → []NoteReadModel        │
├─────────────────────────────────────────────┤
│                                              │
│  実装A: PostgreSQL（キャッシュテーブル）  ← 今回│
│    └─ SELECT * FROM note_read_models ...     │
│                                              │
│  実装B: Redis（将来必要になったら）            │
│    └─ redis.Get("note:" + id)                │
│                                              │
│  → インターフェースが同じだから               │
│    UseCase層は一切変更なし！                  │
│    差し替えるのはFactory層だけ！              │
└─────────────────────────────────────────────┘
```

**これがClean Architectureの力です。**

## 🎯 まとめ

| 質問 | 答え |
|-----|------|
| Read Modelの置き場は何種類？ | 主に4つ（キャッシュテーブル、Redis、Firestore、Elasticsearch） |
| 一番よく使われるのは？ | キャッシュテーブル（コスト0、シンプル） |
| Redisはいつ使う？ | 大規模になってDBの負荷が限界になったとき |
| Firestoreはいつ使う？ | リアルタイム同期やフロント直接アクセスが必要なとき |
| Elasticsearchはいつ使う？ | 全文検索・あいまい検索が重要なとき |
| このプロジェクトでは？ | キャッシュテーブル（学習に最適 + 現場で一番多い） |
| あとから変えられる？ | Clean Architectureだからインターフェースの裏側を差し替えるだけ |
